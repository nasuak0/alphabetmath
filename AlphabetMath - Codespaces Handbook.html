What everything is, in plain words

You are about to use a few tools with unfamiliar names. None of them are complicated once you know what each one is for. Read these once and refer back whenever a word confuses you.

GitHub a website that stores your project safely on the internet and keeps a copy of every version you save. Your project's storage spot there is called a repository — or "repo" for short. Yours is named alphabetmath.
Codespace a complete workspace for running and editing your project that lives on GitHub's computers and opens right inside your web browser. You install nothing, and it does the hard work — not your computer. This is where you will spend almost all your time.
GitHub Desktop a small free app for your computer. Its only job is to carry your project files up to GitHub the very first time. You use it once, in Section 2, and then rarely again.
The studio where I (your assistant) build the project. When you ask me for "the zip," I am handing you a packaged, up-to-date copy of the project to upload.

Three small words GitHub uses

You will see these three words as buttons. Here is all they mean:

  • Commit = save a snapshot and give it a short name (like naming a save file in a game).
  • Push = send your saved snapshots up to GitHub on the internet.
  • Pull = the opposite of push: bring the newest version down from GitHub into wherever you are working.

The one rule that keeps things from breaking

Rule: the project has two folders that sit next to each other — AlphabetMath - App and AlphabetMath - Mobile. One relies on the other, so they must always stay together. Whenever you upload, upload the whole project, never just one folder. (You don't have to think about this — following the steps below does it for you.)

Put the project online (do this once)

The goal of this section: get the project files onto GitHub. This is the only part that uses your own computer, and you never repeat it. Set aside about 20 minutes. You need a free GitHub account and the empty alphabetmath repository (already created).

  1. Install GitHub Desktop. Go to desktop.github.com, download it, install it, and sign in with your GitHub account.
  2. Bring the empty repository onto your computer. In GitHub Desktop, click File → Clone repository. In the list, choose alphabetmath. Choose a place to keep it (for example a new folder called C:\dev\). Click Clone.
    • This creates an empty folder on your computer that GitHub Desktop now watches for changes. Leave it where it is.
  3. Ask me for the zip. Type "give me the zip." I will hand you a single packaged file. Save it and unzip it (right-click → Extract / Unzip). You now have a folder full of the project's files.
  4. Copy the project files into the watched folder. Open the unzipped folder, select everything inside it, and copy it into the folder you cloned in step 2.
    • Copy all of it — the project automatically leaves out anything that shouldn't be uploaded, so you never have to pick and choose.
    • Once copied, you can delete the zip. It has done its job.
  5. Save a snapshot (commit). Switch back to GitHub Desktop. It now shows a long list of files it noticed. In the box at the bottom-left, type a short name like first upload. Click Commit to main.
  6. Send it to the internet (push). Click Push origin at the top. When it finishes, open github.com, go to your alphabetmath repository, and refresh — your files are now there. Section 2 is finished forever.
Quick check before step 5: open the folder you copied everything into. It should contain at least these names. If you see them, you copied the right thing: 
AlphabetMath - App/
AlphabetMath - Mobile/
interactives/
STATUS.md
(plus a few other files)
If you only see one folder, you copied a level too deep — go back into the unzipped folder, open it, and copy what's inside it instead.

Open and run the project

From here on, everything happens in your web browser. The first time takes about 10 minutes (it sets itself up); after that it opens in about a minute.

First look inside: when the workspace opens it can feel busy. There are really just three areas. Down the left is the list of your files and folders. The large middle is where a file's contents show when you click one. The strip along the bottom is the terminal — the text box you type commands into. That's all you need to recognize.
  1. Start a Codespace. On your alphabetmath page at github.com, click the green Code button → the Codespaces tab → Create codespace on main. Wait a couple of minutes the first time while it prepares itself. A code workspace opens in your browser.
    • Next time, you reopen it from the very same menu — it will already exist and open quickly.
  2. Open a command box (terminal). In the top menu of the workspace, click Terminal → New Terminal. A text box appears at the bottom. This is where you type the two lines below.
  3. Type these two lines, pressing Enter after each: 
    cd "AlphabetMath - App"
npm run dev
    • The first line moves into the right folder. The second starts the project.
    • A small pop-up appears saying "Open in Browser." Click it — the AlphabetMath home screen loads in a new tab.
  4. When you're done. To stop the project, click inside the terminal and press Ctrl + C. To leave entirely, just close the browser tab — the Codespace goes to sleep on its own and costs nothing while asleep.
Every time after the first: you don't repeat the setup. Just (1) go to your repository at github.com, (2) click the green Code button → Codespaces and reopen the one already listed, then (3) open a terminal and run the same two lines from step 3. That's it — about a minute.
Why so easy? The project was set up ahead of time so the Codespace arrives with everything it needs already installed. That is why there are no error messages or extra installs here.

Make a public link to share

This gives you a normal web address that anyone can open — no account needed on their end. You set it up once; after that it updates itself.

  1. Turn it on (one time). On your repository page, click Settings → Pages (in the left-hand list). Under Source, choose GitHub Actions. That is the entire setup.
  2. Let it build. Go to the Actions tab of your repository, find "Deploy to GitHub Pages," and if it isn't already running, click Run workflow. Wait for the green check mark (about 2 minutes).
  3. Share your link. Your address is: 
    https://<your-username>.github.io/alphabetmath/
    Replace <your-username> with your GitHub username. It also appears on the Pages settings page. Every future update refreshes this link automatically.
Note: on a free GitHub account, the public link requires the repository to be set to public. If it isn't, open Settings → General, scroll to the bottom, and switch it to public. The private, behind-the-scenes files still never get shared.

What happens after you push a change

Once the link exists, you never set it up again. From then on the site keeps itself up to date on its own. Here is the exact chain each time a change is pushed:

  1. The change is pushed. Your saved edit travels up to the repository on GitHub. (In Phase 1 this is the zip I give you; later it will be your own edits.)
  2. GitHub rebuilds the site by itself. A built-in instruction file tells GitHub: "whenever anything is pushed, rebuild the site and republish it." You press nothing — the push alone starts it. You can watch it on the repository's Actions tab: a spinning dot while it works, then a green check when it's done (about 1–2 minutes).
  3. The link shows the new version. When that green check appears, your address https://<your-username>.github.io/alphabetmath/ is now serving the update. The link itself never changes — only what it shows.
Two things to expect. It is not instant — give it a couple of minutes after the green check. If you still see the old version, your browser is showing a saved copy: press Ctrl + Shift + R to force a fresh load. And if the check turns into a red X, the build failed — the live site simply stays on the last good version (nothing breaks); open the Actions tab and send me what it says.

Get the latest updates from me

While I am still the one building the project, every update travels the same short path. Here it is start to finish:

1. Ask me for a fresh zip.
2. Unzip it and copy everything into your watched folder (say "yes / replace" if it asks).
3. In GitHub Desktop: type a short name, click Commit to main, then Push origin.
4. In your open Codespace: click the Pull (sync) arrow, then run it again (Section 3, step 3).

That's the whole cycle, and it repeats each time there's something new. If the Codespace ever says it is "behind," clicking that same sync arrow (bottom-left of the workspace) fixes it.

If something goes wrong

These are the few snags that actually come up, with the fix for each. Find your situation on the left.

  • “Push origin is greyed out / won't click.” — You haven't saved a snapshot yet. Do the commit first: type a short name in the bottom-left box and click Commit to main. Push will wake up.
  • “The terminal printed a lot of red text after npm run dev.” — If the app still opened in a tab, ignore it. If nothing opened, copy the red text and send it to me.
  • “The Codespace is stuck, blank, or won't open.” — Close the tab and reopen it from Code → Codespaces. If it's still stuck, delete it (the ⋯ menu next to its name → Delete) and create a fresh one. Your files are safe on GitHub — nothing is lost.
  • “My public link shows 404 / page not found.” — One of three things: the Pages source isn't set to GitHub Actions (Section 4, step 1); the repository is still private (make it public); or the build hasn't finished yet (check the Actions tab for a green check mark).
  • “Copying the files asks about replacing or conflicts.” — Choose replace / overwrite. The new files are the newer version — that's exactly what you want.
  • “It says my branch is behind.” — Click the small sync/refresh arrow at the bottom-left of the Codespace. That pulls the newest version down.

Where the code lives (for when you're ready)

You don't need this to run the project — only when you want to start changing it. The whole app is built from one folder: AlphabetMath - Mobile/zmath/. Everything else just reads from it.

The only rule: only change files inside that zmath/ folder. The other folders simply run what's in it.

Which file to open first

Listed easiest-and-safest first:

  • data.js — start here. All the words: topic names, project titles, descriptions, colors. Change a title, save, and it changes on screen. You can't break the layout from this file.
  • tokens.css — the look. All the colors, fonts, and roundness in one place. Change one value and it updates across the whole app.
  • app.css — the detailed styling of each screen.
  • app.jsx — the arrangement of the five screens (Home, Topics, Projects, Notes, About).
  • components.jsx — the reusable parts (the side menu, the tiles, pop-up boxes).
  • Everything else — leave it alone for now; you don't need it to begin.

Three tiny changes to try first

Do them one at a time, saving and looking after each. They prove you know where words, colors, and sizes live.

  1. Change a word. In data.js, edit a topic name or project title. Save. Watch it change.
  2. Change a color. In tokens.css, find --paper and set it to something obvious like #ffe8e8. Save — the background shifts. Then change it back.
  3. Change a size. In tokens.css, make the font-size of .t-h1 bigger. Save.

Making an edit yourself

Changing a file in the Codespace is the same five steps every time, for any file — the app's code, or even this handbook. Learn it once and it never changes.

  1. Open the file. Click it in the file list on the left; its contents fill the middle. (The app's files worth changing live in the zmath/ folder — Section 7.)
  2. Find the text. Press Ctrl + F, type a few words of what you want to change, and it jumps you there.
  3. Type your change. Click into the text and edit it like any document.
  4. Save. Press Ctrl + S. The dot on the file's tab disappears when it's saved.
  5. See it. If the project is running (Section 3), its tab refreshes on its own. For the handbook, right-click the file → Open Preview.

Copying a change I made, without a zip

For small changes you don't need a fresh zip — you can just repeat my edit by hand. Whenever I change something, ask me “show me the lines you changed.” I'll always answer the same way: the file to open, a few words to search for, the old text, and the new text to put in its place. Then you do the five steps above — open, search, swap old for new, save. The result is identical to mine.

One caution: while I'm still building the project, treat your Codespace edits as practice — if we both change the same file, the versions can clash next time I send a zip. When you're ready to take over for real, tell me and we'll hand it over cleanly so you're the only editor.
